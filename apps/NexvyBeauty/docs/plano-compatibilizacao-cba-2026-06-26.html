3. Princípio de design (consenso dos 10)

  • Receita, não builder. Resposta pronta, não canvas "if A then B" — 6 receitas pré-configuradas.
  • Cliente, não lead. Eixo = cliente real (agendamento/pacote/aniversário), não lead de pipeline.
  • Ação = 1 clique → WhatsApp (disparo real via Evolution, já existe).
  • Meta do salão, não quota de SDR. Sem leaderboard; meta é do salão/profissional.
  • Linguagem de salão. "Clientes sumidas", "Onde está o dinheiro parado?".

4. Ordem recomendada (valor × reuso)

  1. E1 + A (fazem par — A consome os cards do E1). Coração da retenção, maior valor/reuso. Começa aqui.
  2. D (Limpeza de Base) — rápido; base limpa destrava WhatsApp/automação.
  3. B (Receitas de Automação) — escala o A de manual → automático.
  4. C (Meta do Mês) — fecha o loop gerencial.

5. Riscos & decisões

  • Não duplicar tabelas. NBA/Scoring já têm lead_nba_sugestao/lead_scoring (pensados pra lead). Estender com contexto de cliente (servico_catalogo.categoria+agendamentos) em vez de criar tabela espelhada.
  • tag_automations tem eventos de pagamento. Estender o enum com eventos de salão (aniversario_cliente, pacote_expirando, retorno_Nd, agendamento_confirmado_24h) — não criar tabela nova.
  • Higiene: merge nunca automático. Diff + confirmação (alguns "duplicados" são parentes com mesmo telefone).
  • Automação opt-in. Preview + contagem antes de disparar. Nunca dispara sozinho sem o 1º opt-in.
Próximo passo: aprovar → implementar na ordem acima, uma feature por vez (mais surfacing que build). Cada uma: migration leve (estender enums) → UI reframada → build → deploy → click-test. Começo pelo par E1 + A no seu GO.
