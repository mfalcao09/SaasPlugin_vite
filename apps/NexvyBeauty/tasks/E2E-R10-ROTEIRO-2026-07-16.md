## Mapa da cadeia (o que o código realmente faz)

```
[Comprador paga R$10]  →  https://pay.cakto.com.br/34h7jqp
        │
        ▼
[Cakto dispara webhook]  →  edge  cakto-webhook  (?scope=platform, ?secret=<webhook_secret>)
        │   valida cred (cakto_credentials.scope='platform') + webhook_secret
        │   upsert em cakto_orders  (mapCaktoOrderForUpsert)
        ▼
[provisionFromOrder]  (_shared/cakto-plan-provisioning.ts)
        │   gate: status ∈ {paid, approved}
        │   resolve plano por cakto_offer_slug='34h7jqp' → fallback cakto_product_id
        │   cria/acha organizations por cakto_customer_email  (ou PROMOVE org demo)
        │   organizations.plan_status='active', plan_id, enabled_modules, demo_expires_at=null
        │   billing_history (amount=10, status='paid', metadata.cakto_id)  ← idempotente
        │   ensureAdminUser → auth user + profiles + user_roles(admin) + email welcome
        │   [se org_created||promoted] seedSalonDataForNewOrg:
        │        servico_catalogo(10) · salon_automation_rules(4, enabled=TRUE) · opportunity_scan_schedules(Radar)
        │        handoffConversationToOnboarding (gated ONBOARDING_HANDOFF_ENABLED, default OFF)
        │        senão → sendWelcomeWhatsApp (genérico)
        ▼
[Onboarding]  welcome-admin-access → send-transactional-email → fila transactional_emails
        │        process-email-queue drena (envia via Resend só se EMAIL_SEND_ENABLED=true)
        │   Wizard de implantação (apply-onboarding, 9 passos) é SEPARADO — o admin roda depois de logar.
        ▼
[Automação]  salon-automation-run consome as 4 regras salon_automation_rules (já enabled=TRUE)
```

**Fatos do código que mudam o teste:**
- **Provisionamento só ocorre no `scope=platform`.** Tag-automations, agente de recuperação (`cakto-recovery-trigger`) e resolução de lead **só rodam em `scope=organization`** → **não** procure por eles neste teste.
- O e-mail welcome é **sempre enfileirado** (`email_send_log.status='pending'`), mas só **sai** via Resend se `EMAIL_SEND_ENABLED=true` (senão fica `dry_run`).
- As 4 receitas de automação nascem **`enabled=TRUE`** (decisão P9/CART 2026-07-15) — não dependem de flag.
- `billing_history.amount` recebe `order.amount` **como veio da Cakto**. O código assume **reais** (10), não centavos. Se vier 1000, é bug de unidade (ver Risco R6).

### 1.3 — O segredo do webhook confere? (esta é a "assinatura" da Cakto)
> A Cakto **não** manda HMAC. A autenticação é um **segredo compartilhado** que a Cakto envia
> na URL (`?secret=…`) ou no header `x-cakto-secret`, comparado a `cakto_credentials.webhook_secret`.
- **Ação:**
```sql
select scope, organization_id,
       (webhook_secret is not null and webhook_secret <> '') as tem_secret
from cakto_credentials
where scope = 'platform';
```
- **Esperado:** 1 linha, `scope='platform'`. Se `tem_secret = true`, a URL do webhook cadastrada no
  painel da Cakto **precisa** conter esse mesmo `?secret=…` (ou o header). Se `tem_secret = false`,
  o webhook aceita qualquer chamador (sem validação de segredo).
- **Verificar (lado Cakto):** no painel da Cakto, a URL de webhook do produto deve ser
  `…/functions/v1/cakto-webhook?scope=platform&secret=<valor_de_webhook_secret>`.
- **PASS/FAIL:** ✅ segredo do painel == `webhook_secret` do banco (ou `tem_secret=false`) ·
  ❌ divergente → webhook retorna **401** e dispara alerta Telegram "assinatura inválida" (venda paga **sem** provisionar).

### 1.4 — O webhook está acessível SEM JWT?
> `cakto-webhook` **não** está listado em `supabase/config.toml`; depende de ter sido deployado
> com `--no-verify-jwt`. Se alguém redeployar sem a flag, o gateway passa a rejeitar TODA chamada
> da Cakto com 401 **antes** do código rodar (nenhum alerta Telegram — o código nem executa).
- **Ação:** `curl` sem Authorization:
```bash
curl -i -X POST "https://fzhlbwhdejumkyqosuvq.supabase.co/functions/v1/cakto-webhook?scope=platform" \
  -H "Content-Type: application/json" -d '{}'
```
- **Esperado:** resposta do **código** — HTTP `404 {"error":"credentials not found"}` **ou**
  `400 {"error":"invalid payload"}` **ou** `401 {"error":"invalid secret"}`.
- **PASS/FAIL:** ✅ qualquer uma dessas (o código rodou) ·
  ❌ `401 {"message":"Missing authorization header"}` (erro do **gateway**) → o webhook está exigindo JWT.
  **PARE e redeploye** `cakto-webhook` com `--no-verify-jwt`.

  `opportunity_scan_schedules`, `servico_catalogo`/products, etc.) → grava `platform_audit_logs`.

### 7.2 — Verificar que a org sumiu
- **Ação:**

### 7.3 — Resíduos que NÃO são apagados pela cascata (limpar à mão)
A `delete-organization` cobre a org e seus filhos por FK, mas alguns resíduos **não** têm FK para `organizations`:

- **`cakto_orders` (scope `platform`)** — tem `organization_id = NULL` no escopo platform → **não** cascateia. Apagar:
```sql
-- Confirme antes com SELECT; execute o DELETE só após conferir o cakto_id.
delete from cakto_orders
where scope = 'platform'
  and lower(customer_email) = lower('<EMAIL_TESTE>')
  and cakto_id = '<CAKTO_ID_DO_PASSO_3.2>';
```
- **Usuário no Auth** — se você for **reusar o mesmo e-mail** num próximo teste e o `delete-organization`
  já removeu o usuário (ele remove os `profiles` da org), confirme:
```sql
select count(*) from profiles where lower(email) = lower('<EMAIL_TESTE>');  -- esperado 0
```
  Se ainda houver usuário órfão no Auth, remova pelo Dashboard → Authentication → Users.
- **`email_send_log` / `email_unsubscribe_tokens` / `suppressed_emails`** — logs de e-mail **não** cascateiam.
  São inofensivos (histórico), mas se for reusar o e-mail e ele tiver caído em `suppressed_emails`,
  remova essa linha (senão o próximo welcome é suprimido). `email_send_log` pode ficar como histórico.
- **`platform_audit_logs`** — a linha `Empresa excluída permanentemente` fica **de propósito** (trilha de auditoria). Não apague.
