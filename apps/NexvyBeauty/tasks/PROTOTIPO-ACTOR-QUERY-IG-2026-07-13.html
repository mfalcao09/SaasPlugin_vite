1. Veredito

VIÁVEL — porém a premissa central estava errada e precisa ser corrigida.
  • [Certo] O endpoint de busca dentro dos seguidores existe e aplica o filtro query server-side (só volta quem casa). A tese "pagar só pelo rio, não pelo oceano" se sustenta na banda.
  • [Certo] Ele NÃO é guest. Exige sessão autenticada (sessionid). Provado ao vivo: guest devolveu HTTP 401. O briefing dizia "o endpoint guest não usa login" — incorreto para followers: a lista fica atrás do login-wall independente do query.
  • Consequência: produção precisa de proxy residencial + POOL DE CONTAS LOGADAS — não só proxy. Esse é o custo/risco binding, não a banda.
  • Não provei daqui: que o filtro devolve só-matches ao vivo — exige sessão válida que não tenho/não devo criar no spike. Confirmação do filtro é [Provável], ancorada no código do instagrapi.
Correção de premissa (Modo Conselheiro): o diferencial do nosso actor vs. o de mercado (jWD4G57HhqYY0mFhd) não é acesso guest (não existe) — é empurrar o query pro servidor e paginar só o subconjunto casado, usando as mesmas contas logadas que qualquer scraper de followers já precisa. O ganho é de banda/páginas trafegadas, não de dispensar auth.

2. Endpoint confirmado + fontes

2.1. Resolução username → user_id (guest, funciona)

GET https://i.instagram.com/api/v1/users/web_profile_info/?username=<username>
Header obrigatório:  x-ig-app-id: 936619743392459   (web app id do IG, público)

Retorna data.user.id (+ edge_followed_by.count). Guest-acessível com só o x-ig-app-id. Provado ao vivo (§3).

2.2. Busca dentro dos seguidores — o "rio" (autenticado)

GET https://www.instagram.com/api/v1/friendships/<user_id>/followers/
      ?count=<n>
      &search_surface=follow_list_page
      &query=<keyword>          <-- filtro server-side: só volta quem casa
      &enable_groups=true
      [&max_id=<cursor>]        <-- paginação (next_max_id do corpo anterior)
Headers:  x-ig-app-id: 936619743392459
          x-csrftoken: <csrftoken>
          cookie: sessionid=<...>; csrftoken=<...>
  • Params search_surface=follow_list_page + query + enable_groups=true vêm verbatim do search_followers_v1 do instagrapi (a caixa de busca do app usa essa superfície).
  • Host web www.instagram.com/api/v1/... ≡ mobile i.instagram.com/api/v1/friendships/<id>/followers/. Ambos private_request() = autenticado.

4. Diagnóstico de bloqueio (honesto)

Dois eixos independentes:

  1. Auth (bati agora): followers exige sessionid. Guest = 401 sempre. Proxy não resolve — só conta logada.
  2. IP / rate (não testei): com sessão, IG bane IP datacenter "instantly" e limita ~200 req/h/IP. Aqui entra o proxy residencial com sticky 5–10 min.

Para medição representativa faltam as duas coisas juntas — sessão logada e proxy residencial. Deste ambiente isolei só o eixo (1).

  1. 1 conta IG logada descartável (aged, não a da dona) → extrair sessionid + csrftoken dos cookies pós-login. Passar via env, nunca hardcode: 
    IG_SESSIONID=<...> IG_CSRFTOKEN=<...> \

    O protótipo já lê essas envs → "modo AUTENTICADO". Isso já prova o filtro (esperado: 200 + só usernames com "manicure").
  2. Proxy residencial (Apify Residential / IPRoyal / Bright Data) sticky 5–10 min. Standalone: undici.ProxyAgent/HTTP_PROXY. Actor: Actor.createProxyConfiguration({ groups:['RESIDENTIAL'], countryCode:'BR' }) passado ao fetch/dispatcher.
  3. Wrapper Apify Actor: trocar argv() por Actor.getInput(), envolver main() em Actor.main(), persistir via Actor.pushData(). Core (resolveUserId + searchFollowersPage) intacto.
  4. Medição: paginação real (5–10 páginas), coletar GB do proxy (Apify reporta por run) ÷ matches → $/match empírico. 2–3 contas → taxa de challenge/ban (o número que decide a economia).

Gate de decisão: medir a taxa de ban por conta por N requests antes de escalar. Conta aguenta milhares de queries → modelo fecha barato. Queima em centenas → custo de conta domina e o $/match sobe 1–2 ordens acima da banda.
