Fatos do código que mudam o teste:

  • Provisionamento só ocorre no scope=platform. Tag-automations, agente de recuperação (cakto-recovery-trigger) e resolução de lead só rodam em scope=organization → não procure por eles neste teste.
  • O e-mail welcome é sempre enfileirado (email_send_log.status='pending'), mas só sai via Resend se EMAIL_SEND_ENABLED=true (senão fica dry_run).

1.3 — O segredo do webhook confere? (a "assinatura" da Cakto)

A Cakto não manda HMAC. A autenticação é um segredo compartilhado enviado na URL (?secret=…) ou no header x-cakto-secret, comparado a cakto_credentials.webhook_secret.
select scope, organization_id,
       (webhook_secret is not null and webhook_secret <> '') as tem_secret
from cakto_credentials
where scope = 'platform';
  • Esperado: 1 linha, scope='platform'. Se tem_secret=true, a URL do webhook no painel Cakto precisa conter o mesmo ?secret=…. Se false, aceita qualquer chamador.
  • Verificar (lado Cakto): URL deve ser …/functions/v1/cakto-webhook?scope=platform&secret=<valor>.
  • ✅ segredo do painel == banco (ou tem_secret=false) · ❌ divergente → webhook retorna 401 + alerta Telegram (venda paga sem provisionar).

1.4 — O webhook está acessível SEM JWT?

cakto-webhook não está em supabase/config.toml; depende de ter sido deployado com --no-verify-jwt. Um redeploy sem a flag faz o gateway rejeitar TODA chamada Cakto com 401 antes do código (nenhum alerta — o código nem executa).
curl -i -X POST "https://fzhlbwhdejumkyqosuvq.supabase.co/functions/v1/cakto-webhook?scope=platform" \
  -H "Content-Type: application/json" -d '{}'
  • Esperado: resposta do código — 404 {"error":"credentials not found"} ou 400 {"error":"invalid payload"} ou 401 {"error":"invalid secret"}.
  • ✅ qualquer uma dessas (o código rodou) · ❌ 401 {"message":"Missing authorization header"} (gateway) → PARE e redeploye com --no-verify-jwt.
