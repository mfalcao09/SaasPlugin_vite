TL;DR

O pipeline de e-mail (sender + fila pgmq + cron + consumidor + branding NexvyBeauty) já existia e já estava deployado (PR #61). O e-mail não saía por um motivo real, agora corrigido:

  • Bug do 403 (corrigido): o cron autentica com a chave de service-role do vault, que é a chave nova opaca sb_secret_… (não é JWT). O check parseJwtClaims(token).role==='service_role' falhava → 403 a cada minuto → a fila nunca era consumida (1 e-mail preso há ~7 dias). Agora a auth aceita sb_secret_ e o JWT legado.
  • Gate seguro (novo): o envio real exige um switch EXPLÍCITO. Default = DRY-RUN (drena a fila, registra dry_run, não envia). O fix do 403 não liga envio real sozinho.

⚠️ Verificado: a RESEND_API_KEY JÁ ESTÁ setada em prod (o dispatcher respondeu dry_run:false antes do gate seguro). O "gate na chave" que o briefing assumia pendente já foi feito. Falta 1 flag + confirmar DNS.

✅ Ação do Marcelo — checklist de go-live (nesta ordem)

  1. Confirmar o domínio nexvybeauty.com.br VERIFICADO na Resend (Dashboard → Domains). Precisa dos 3 registros DNS verdes:
    • SPF (TXT) — include:amazonses.com / o que a Resend indicar (região sa-east-1).
    • DKIM — os CNAME resend._domainkey… que a Resend gerar.
    • DMARC (TXT em _dmarc.nexvybeauty.com.br) — começar v=DMARC1; p=none; rua=mailto:….
    Sem domínio verificado, a Resend responde 403 (domain not verified) → a mensagem vai para a DLQ (não entrega). Verificar DNS antes de ligar o envio.
  2. Ligar o envio real (a RESEND_API_KEY já existe): 
    supabase secrets set EMAIL_SEND_ENABLED=true --project-ref fzhlbwhdejumkyqosuvq
    No minuto seguinte o cron passa a enviar de verdade (dry-run desliga sozinho).
  3. Validar go-live (1 e-mail real de teste): dispare um e-mail real e confira: 
    select status, recipient_email, template_name, created_at
from public.email_send_log
where created_at > now() - interval '10 minutes'
order by created_at desc;
    Esperado: status = 'sent' (não dry_run/dlq/failed).
  4. (Opcional) Limpar a DLQ: há 1 mensagem presa há ~7 dias em transactional_emails_dlq (o e-mail travado pelo 403; expirou por TTL e foi para a DLQ ao rodar o dispatcher). 
    select * from pgmq.read('transactional_emails_dlq', 30, 10);   -- inspecionar
select pgmq.purge_queue('transactional_emails_dlq');            -- limpar

⏪ Rollback / desligar o envio

# Volta para DRY-RUN (não envia; drena e registra 'dry_run'):
supabase secrets unset EMAIL_SEND_ENABLED --project-ref fzhlbwhdejumkyqosuvq
# ou forçar dry-run mesmo com o flag ligado:
supabase secrets set EMAIL_DRY_RUN=true --project-ref fzhlbwhdejumkyqosuvq

Comportamento do gate

Envio real exige, cumulativamente: 1. RESEND_API_KEY presente (já setada) 2. EMAIL_SEND_ENABLED=true (pendente) 3. EMAIL_DRY_RUN ≠ true

Qualquer condição faltando → DRY-RUN: lê a fila, registra dry_run e remove da fila (drena), sem chamar a Resend.

⚠️ Em DRY-RUN, e-mails reais (auth/pós-compra) são registrados como dry_run e NÃO entregues. Ligue o flag (passo 2) no go-live, com o DNS verde. Antes desta entrega esses e-mails também não saíam (o 403 travava tudo) — a diferença é o switch limpo e o cron parar de quebrar.
