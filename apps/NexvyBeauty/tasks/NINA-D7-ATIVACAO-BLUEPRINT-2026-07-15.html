1. A RESPOSTA do D-7 (âncora + SQL exato)

1.1 O que dá pra afirmar

CERTO Dá para computar "faltam 7 dias" — duas âncoras, uma frágil e uma correta:

ÂncoraFonteConfiabilidadeCusto
(A) plan_activated_at mod 30 == 23 (o scan hoje)organizations.plan_activated_atMédia — exata só p/ mensal-30zero (já codado)
(B) next_payment_date − 7cakto_orders.raw_payload->subscriptionAlta — data-verdade Cakto; imune a ciclo/anual/dunningpersistir coluna + trocar computeRenewalPosition

1.2 Por que (A) funciona — e onde quebra

CERTO A Cakto usa período fixo de 30 dias (recurrence_period: 30; no payload de teste paid_at 2026-07-11 → next_payment_date 2026-08-10 = +30d). E cakto-plan-provisioning.ts:209 grava plan_activated_at = now() em todo webhook pago (inclusive renovação — org achada por cakto_customer_email). Logo, para mensal, mod 30 == 23 acerta 7 dias antes de cada renovação sem depender de nada externo.

CERTO Onde quebra:

  • Planos anuais. platform_plans tem price_monthly e price_yearly. Oferta anual → recurrence_period=365, mas o env NINA_RENEWAL_CYCLE_DAYS é global (default 30) → dispara ~12x/ano errado.
  • plan_activated_at de seed manual. A única org ("Studio Bella (Teste)", 2026-06-18) não veio da Cakto (cakto_subscription_id NULO) → mod 30 vira âncora arbitrária.
  • Dunning/retry (max_retries:3): renovação atrasada re-ancora e defasa o próximo next_payment_date.

1.3 Recomendação

PROVÁVEL Âncora recomendada: (B), antes da base crescer. Persistir next_payment_date+recurrence_period no provisionamento (dado já chega no raw_payload) e computar D-7 = next_payment_date::date − current_date = 7. Resolve anual e dunning de graça.

CERTO Para ligar JÁ (v1): (A) é defensável — se (i) nenhuma oferta ativa for anual e (ii) a base real for ~0 (é: 1 org de teste). Hoje o scan alcança 0 contas (days_since=27, 27 % 30 = 27 ≠ 23). Risco do D-7 hoje = nulo; o risco real é o Fato 4.

1.4 SQL exato

(A) — verificação do modelo atual (quem o scan pegaria hoje):

select o.id, o.name, o.plan_activated_at,
       floor(extract(epoch from (now() - o.plan_activated_at))/86400)::int      as days_since,
       floor(extract(epoch from (now() - o.plan_activated_at))/86400)::int % 30 as cycle_pos
from public.organizations o
where o.plan_status = 'active'
  and o.plan_activated_at is not null
  and floor(extract(epoch from (now() - o.plan_activated_at))/86400)::int % 30 = 30 - 7;  -- targetPos=23
-- Resultado hoje (2026-07-15): 0 linhas.

(B) — âncora recomendada, lendo a data-verdade já existente:

-- Quando next_payment_date for persistido (ex.: organizations.next_renewal_at):
--   select id, name from public.organizations
--   where plan_status='active' and (next_renewal_at::date - current_date) = 7;

-- Enquanto não persistido, a fonte crua (por org, último pedido):
select distinct on (co.organization_id)
       co.organization_id,
       (co.raw_payload->'subscription'->>'next_payment_date')::timestamptz as next_payment_date,
       (co.raw_payload->'subscription'->>'recurrence_period')              as recurrence_period
from public.cakto_orders co
where co.scope = 'organization'
  and co.raw_payload->'subscription' is not null
order by co.organization_id, co.paid_at desc;
-- Hoje: 0 linhas úteis (único pedido é scope='platform', organization_id NULL, dado de teste).
