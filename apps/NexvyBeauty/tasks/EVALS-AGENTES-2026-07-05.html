EVALS-AGENTES — a régua binária do "95/100"

2026-07-05 · NexvyBeauty · brain de vendas platform-sales-brain (Duda → Bia) · braço EVALS-V1 (5.6)
sem eval, nenhuma nota se afirma
O gate: agente aprovado = ≥ 90% das asserções passando (summary.gate_90pct === true). Cada golden só "passa" se todas as suas asserções passam. gate_90pct é o sinal binário de release.

1. Arquitetura

POST /functions/v1/tmp-eval-agents   (header: x-brain-secret)
  body: { golden_id? , only?[] , keep? , cleanup? }
   │
   ├─ resolve produto (slug 'nexvybeauty') + closer (Bia) ativo no WhatsApp
   ├─ cleanup defensivo de resíduos 'wa:eval-%'
   │
   └─ para cada GOLDEN:
        1. cria conversa efêmera  visitor_id = 'wa:eval-<id>-<rand>'
           + lead efêmero (metadata.eval=true; semeia qualificação se leadSeed)
        2. injeta inbound[]  (wa_timestamp = agora − 35s → derrota o debounce)
        3. chama platform-sales-brain  { conversation_id }
        4. lê as bolhas outbound persistidas → roda as ASSERTIONS binárias
   │
   ├─ cleanup final (deleta 'wa:eval-%' + leads eval)  [pula se keep=true]
   └─ retorna { summary, goldens[] }

Arquivos (em supabase/functions/tmp-eval-agents/):

  • goldens.ts — as 12 golden conversations (input + asserções + seed).
  • assertions.ts — motor binário puro (regex / contagem de '?' / link).
  • index.ts — runner: cria efêmeros, injeta, chama o brain, coleta, pontua, limpa.
  • ../../config.toml — entry [functions.tmp-eval-agents] verify_jwt = false.

1.1 Por que ZERO mensagem real é enviada (defesa em profundidade)

O brain entrega no WhatsApp Cloud API via deliverViaWhatsAppCloud, que faz to = String(toPhone).replace(/\D/g,'') e aborta com no_destination_phone se to fica vazio. A EF cria a conversa com visitor_whatsapp = 'eval-no-send' (sem nenhum dígito) → o Graph nunca é chamado. Mesmo com uma conexão Cloud API active no ambiente, nada sai. As bolhas ainda são persistidas no CRM (o brain persiste antes de entregar) — é de lá que o eval lê a resposta.

1.2 Por que o debounce não trava o eval

O brain tem DEBOUNCE_MS = 25s: se a última inbound é fresca, ele dorme. A EF injeta as inbounds com wa_timestamp = agora − 35s (além do debounce, mas abaixo do STALE_REDELIVERY = 10min) → o brain responde na hora. Exceção proposital: o golden g_mensagens_picadas_debounce usa timestamps frescos (1–3s) para provar que a rajada agrega numa resposta só.

1.3 Memória de qualificação (leadSeed)

Goldens que exigem lead já qualificado (decidido, cético, pede desconto) trazem leadSeed → a EF grava platform_crm_leads.metadata.qualificacao + bant_*/temperature antes de chamar o brain, reproduzindo o estado real. A Duda não reperguntar o que já sabe.

2.1 Vocabulário nunca-permitido (asserção compartilhada, escopo all)

  • grátis/teste grátis → o sanitizeReply deve ter reancorado na garantia.
  • desconto/promoção → idem.
  • não se encaixa/não se qualifica → desqualificação: proibida.
  • tags de controle ([PASSAR_BIA], [ESCALAR_HUMANO], [HANDOFF_HUMANO]) → nunca no texto ao cliente.

4. Como rodar

Pré-requisitos: SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY, BRAIN_INTERNAL_SECRET, AI_API_KEY/AI_GATEWAY_URL (os mesmos do brain), e o produto nexvybeauty com Duda/Bia ativos no WhatsApp.

Este braço não deploya. Para rodar contra o ambiente real, a EF precisa estar servida (supabase functions serve tmp-eval-agents local, ou deploy pontual sob demanda — fora do escopo deste braço).

Todos os goldens:

curl -sS -X POST "$SUPABASE_URL/functions/v1/tmp-eval-agents" \
  -H "x-brain-secret: $BRAIN_INTERNAL_SECRET" \
  -H "Content-Type: application/json" \
  -d '{}' | jq '.summary'

1 golden (a falha de 05/07):

  -d '{"golden_id":"a_falha_50_clientes"}' | jq '.goldens[0]'

Subconjunto, mantendo as conversas p/ inspeção:

  -d '{"only":["b_decidido_manda_link","c_qualificado_cetico_passa_bia"],"keep":true}'

Só limpar os efêmeros (idempotente):

  -d '{"cleanup":true}'

4.1 O que olhar na saída

  • summary.gate_90pct — o veredito binário (true = aprovado).
  • summary.assertions_pass_rate — a nota real (0..1).
  • summary.goldens_passed / goldens_total — cenários que passaram inteiros.
  • goldens[].failures[] — por asserção reprovada: reason (a regra) + detail (o que a régua viu, ex.: PROIBIDO encontrado: "grátis").
  • goldens[].last_turn — o texto real que a Duda/Bia produziu.
  • goldens[].brain[] — retorno do brain (skipped, handoff, passed_to_bia, bubbles).

6. Limitações honestas

  • LLM real (gemini-2.5-flash) → eval não-determinístico na margem. Asserções miram o comportamento (regex amplo), mas pode haver falso-negativo. Rodar 2× antes de acusar regressão.
  • Custo: ≥2 chamadas LLM por golden (resposta + extração). 12 goldens ≈ 24–30 chamadas/rodada.
  • Depende do banco real: sem public_plans com checkout_url, o golden b_decidido_manda_link falha por falta de dado, não por erro do agente.
  • max_questions conta ? literal — é um piso, não perfeição.
  • A régua é tão boa quanto os goldens. Toda falha nova de produção deve virar um golden aqui.

7. Divergência de preço a conferir (QCR §5)

A âncora da conta é R$217 (Essencial); o extrator do brain usa ÷217 no score. Confirmar no public_plans que o plano de entrada é 217 antes de tratar o score do eval como verdade — o eval mede comportamento (rota, vocabulário, forma), não a aritmética do score.
