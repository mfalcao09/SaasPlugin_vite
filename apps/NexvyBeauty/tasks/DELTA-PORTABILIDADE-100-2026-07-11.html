Pergunta: "75% não é confortável. O que falta para os 100% viável — os 100% do que é possível pro nosso CRM?" · Padrão DURO: se não é cópia fiel, é gap.

0. Veredito honesto

O número raw bate com teu incômodo: ~74% portado 1:1. Mas o delta não é caos espalhado — concentra em 2 blocos grandes + um rabo escasso, e parte dos "20%" é trilha-própria que você já parkou (Voz IA, Meta Ads, fluxos IG), não buraco de porte.

A verdade desconfortável: o maior gap real não é o inbox (esse está fiel) — é o cérebro/conteúdo dos agentes (objeções, materiais, base de conhecimento, catálogo, pós-venda): UI portada 1:1 mas backend inerte (tabelas platform_crm_* product-scoped nunca criadas). E há um cluster de "mentira silenciosa" (UI diz "salvo" e não persiste) que é bug de produto.

2. Como o delta se agrupa

🅰 Trilhas próprias — JÁ parked (não é buraco de porte)

Decisões de produto: Voz IA (10 edges + CallVoiceAIDialog, GRANDE) · Meta Ads/CTWA (marketing-connect/sync, meta-ads-validate) · Fluxos IG (ig-flow-executor, instagram-flow-generate-ai).

🅱 Cérebro/conteúdo do agente — O MAIOR BLOCO (~11-12d)

Config/orquestração/MIA/goals/captação persistem 100%. O gap é o conteúdo product-scoped: UI portada 1:1, mas mutações caem num stub (useProductHubStubs.ts → toast.info + query []). 8 tabelas + ~7 edges twin faltando: base de conhecimento, catálogo de itens, pós-venda, objeções estruturadas, materiais, vídeo-aulas, CTAs do chat, templates de e-mail, treino do agente, gerar-agente-IA, chat de teste.

Causa-raiz única: faltam tabelas platform_crm_* sem organization_id + versão platform-* de edges que já existem org-scoped (reuso trocando organization_id→product_id). Impacto: alimenta a Duda/Bia por produto — coração do "agente que vende".

🅲 Dashboard de Jornada / atribuição (~2-3d)

lib/leadJourney.ts (760 linhas) + useLeadJourney.ts (11 hooks) + 10 componentes journey/ não portados. Funil, heatmap, aquisição por origem/campanha/criativo, gargalos, touchpoints, feed real-time. A aba de jornada por-lead ESTÁ portada; falta o dashboard analítico.

🅳 Inbox borda + "MENTIRA SILENCIOSA" (~2-3d)

Núcleo do atendimento fiel. Mas 3 gaps são bug de produto (UI diz "salvo", não persiste):

  • 🔴 TransferModal — troca de conexão Evolution vira console.warn, não persiste.

  • FollowupAI sem warnings/model · AcceptTicket sem force:takeover · InboxMetricsHeader e ProductSelector não montados · LeadEditModal (editar lead vinculado) sempre abre só o EditVisitor.

🅴 Instagram — sender real de DM (~2-3d) ⚠️ VERIFICAR

platform-instagram-* só tem connect/draft/test/webhook. Faltam instagram-send (sender de DM real), instagram-list-media, instagram-subscribe-fields. ⚠️ Conflito: sessões anteriores marcaram "IG outbound funcionando" — precisa confirmar se o envio real funciona ou passa por outra rota.

🅵 Features de apoio — re-exposição (~1.5-2d)

Tudo existe tenant-side; falta re-expor em platform_crm_*: summaries deals/comissões, gamificação de metas, Google Calendar connect, AICampaignAssistant, squad-performance comparativo.

🅶 Cadência (~2-3d)

Editor + modelo byte-idênticos. Faltam: biblioteca de contextos reutilizáveis (context_id) + editor rico no hub do produto (stub todoBackend).

3. 🔑 O lote de migrations que mata a "mentira silenciosa"

Um padrão atravessa 🅱🅳🅶: a UI existe e mente porque falta a tabela/coluna platform_crm_*. Um lote único de migrations product-scoped (RLS via helper SECURITY DEFINER, padrão já usado) destrava de uma vez: 8 tabelas de conteúdo + visitor_email + colunas do Transfer + context_id de cadência. É o movimento de maior retorno — transforma dezenas de UIs inertes em funcionais e mata os bugs onde o sistema mente pro operador.

6. Ressalva de método

Auditoria por leitura profunda + grep dos dois lados, não diff -r byte-a-byte em cada arquivo. Onde afirmei "byte-idêntico" (useLeadTracking, useCadence) foi confirmado; nos blocos grandes por presença em caminho idêntico + amostra. Rigor total = diff -r nas pastas (~1h).
