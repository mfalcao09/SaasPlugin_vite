3. Modelo de dados — a tabela de pendências

Uma linha = "falta o campo X do cliente Y, e é isto que já fizemos a respeito". É o caderninho do Auditor e o estado do follow-up na mesma tabela ("um registro só").

-- PROPOSTA (não aplicada). Mecanismo dedicado, isolado do fluxo de dinheiro.
create table public.salon_client_field_requests (
  id              uuid primary key default gen_random_uuid(),
  organization_id uuid not null,
  cliente_id      uuid not null references public.clientes(id) on delete cascade,
  campo           text not null check (campo in
                    ('data_nascimento','endereco','email','cpf_cnpj')),
  status          text not null default 'pending' check (status in
                    ('pending',     -- Auditor detectou o buraco
                     'asked',       -- follow-up perguntou (carona saiu)
                     'answered',    -- respondeu, valor extraído e GRAVADO
                     'declined',    -- recusou / opt-out -> não pergunta mais
                     'unreachable', -- sem telefone válido / ambíguo
                     'skipped')),   -- resolvido por outra via (form manual)
  valor_pendente  text,            -- valor bruto extraído, aguardando gravação
  asked_at        timestamptz,
  answered_at     timestamptz,
  ask_count       int not null default 0,   -- cap p/ não insistir
  last_channel    text,
  source_message_id uuid,          -- webchat_messages.id da resposta (trilha)
  extraction_confidence numeric,   -- 0..1 (regex=1.0; LLM=score) -> gate confirmação
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now(),
  -- IDEMPOTÊNCIA: 1 pendência viva por (cliente, campo). Não pergunta 2x.
  unique (organization_id, cliente_id, campo)
);
create index idx_scfr_org_status on public.salon_client_field_requests(organization_id, status);
create index idx_scfr_cliente     on public.salon_client_field_requests(cliente_id);
alter table public.salon_client_field_requests enable row level security;
create policy "org reads scfr" on public.salon_client_field_requests
  for select using (organization_id = get_user_organization(auth.uid()));
  • unique(org,cliente,campo) = idempotência: Auditor faz upsert, nunca duplica, nunca reabre answered/declined.
  • ask_count + cap (máx ~2) evita virar chateação.
  • extraction_confidence decide o gate: regex casou (1.0) → grava direto; LLM 0.6 → confirma antes.
  • source_message_id = trilha (prova de qual mensagem veio o dado — LGPD §4).
  • Grava em clientes.<campo> só no answered: pendência = diário, clientes = verdade.

Políticas gravadas no fluxo (verbatim do briefing):
  • 1 campo por vez — nunca despeja formulário no WhatsApp.
  • Sempre carona — só pergunta pendurado num toque que já ia sair. Zero conversa fria.
  • Idempotência — unique(org,cliente,campo) + status/ask_count: nunca pergunta 2× nem regrava.
  • Confirmação sob dúvida — regex grava direto; LLM inferido confirma antes.

5. Decisões pro Marcelo (com recomendação)

Decisão 1 — Receitas: default ligado vs. opt-in

Enquanto ligar depender do dono, fica 0% (o DB prova). O toggle não é "empoderamento", é o ralo por onde o valor vaza.

OpçãoImplicaçãoLGPD
(a) Opt-in puro (hoje)0% de adoção comprovadoConservador
(b) Ligado por default no onboarding + prévia + kill-switchAutomação de verdade desde o dia 1OK: legítimo interesse + relacionamento existente; dono vê e desliga
(c) Default só as "seguras" (lembrete 24h, pós-atendimento)Meio-termoMais conservador
Recomendação: (b) com salvaguardas. Ligar as 4 receitas por default no apply-onboarding, com a tela mostrando "3 receitas ligadas, ver prévia" + kill-switch visível. O dono nunca precisa ligar; pode desligar. Isso sozinho move de 0% pra funcional. Conservador no lançamento? Comece em (c), promova pra (b) após 1 semana. Risco de (b): base suja → mensagem torta — mitigado pelo Auditor + prévia.

Decisão 2 — Campos prioritários

Ordem: data_nascimento > telefone (já 100%, ignora) > endereco > email > cpf_cnpj.
  • Nascimento primeiro — destrava a receita de maior carinho/ROI (aniversário), hoje em 10%. Maior alavanca.
  • Endereço depois — 0% hoje; só prioriza se há entrega/atendimento em casa. Via CEP (1 pergunta) + BrasilAPI auto-completa o resto (minimização).
  • CPF por último — sensível, alto atrito, baixo uso. Só se emite nota. Não perseguir por carona.

Decisão 3 — O agente de follow-up: Duda / Nina / novo?

Recomendação: nenhum agente-persona novo. Follow-up = motor de receitas (carona) + salon-collect-inbound (captura). NÃO é a Duda.
  • Por que não a Duda: é a IA de vendas da plataforma — atende leads (bot_active, CRM). Domínio errado: cliente-do-salão vive em clientes, não em leads. Plugar aqui repete o erro "Booking CRM ≠ agenda salão".
  • Por que não a Nina (P2): é um dos 3 agentes-casca em construção (prompt vazio). Herdar uma persona conversacional inteira pra perguntar 1 data é canhão pra mosquito + acopla frentes.
  • Por que "sem persona nova" é o certo: o menor loop não precisa de bot conversacional — precisa do motor determinístico anexar 1 pergunta + 1 edge que leia a resposta. É captura estruturada, não diálogo. Fase 2 opcional: se o tenant tiver bot recepcionista ativo (webchat-bot), ele pode perguntar conversacional — enfeite, não MVP.

LGPD — base jurídica do Agente de Carteira

Papéis: salão = controlador; NexvyBeauty = operador. Nascimento + endereço do cliente-do-salão = dado de terceiro — a NexvyBeauty só processa sob instrução.

  • Base legal (Art. 7): data_nascimento → legítimo interesse (VII) apoiado em relacionamento pré-existente (+ teste de proporcionalidade + opt-out fácil). endereco → execução de contrato (V) se há entrega; senão consentimento. cpf_cnpj → obrigação legal/fiscal (II) se emite nota; fora disso, não coletar.
  • Minimização (Art. 6 III): 1 campo por vez; endereço via CEP (pede menos, deduz o resto).
  • Aviso na 1ª interação: pergunta-carona vem com o porquê; evento em lgpd_consents (scope salon_client_field_collection, consent_text exato, source_message_id). Recusa (declined) é honrada, nunca reperguntada.
  • Trilha (Art. 37): source_message_id prova de qual mensagem/quando/qual aviso.
  • Gap a sinalizar: não há coluna de opt-out por-cliente em clientes. declined cobre o campo, mas um "não perturbe" global (Art. 18) precisa de lugar próprio — recomendo flag clientes.marketing_opt_out boolean no PR do F5.
