🔑 P1 — Eval de Conversão: o Keystone da Nave BDR

Detalhamento do P1 de NAVE-BDR-AUTONOMO §4-6. Critério de pronto: golden set 50+ · runner local+CI · falso-descarte bloqueia PR · opt-in/report-rate no dashboard.

0. TL;DR

O eval de conversão é um portão único com quatro clientes: (1) variantes de prompt do sales-spark, (2) o cérebro da Duda/Bia (platform-sales-brain), (3) as skills do Hermes agent (auto-editáveis, SEM gate nativo → deploy dele é pós-P1), e (4) o corpus que o reindex da cerebro-infra reingere diariamente. Nada desses quatro muda em produção sem passar no golden set.
Decisão de design: casos vivem em git (evals/sales/golden/*.jsonl) — CI bloqueia PR sem credencial de banco, caso revisável em diff, mesmo arquivo gate-ia prompt E skill do Hermes. No banco ficam só agent_eval_runs/agent_eval_results (histórico/dashboard).

1.1 "Falso-descarte" redefinido (diretiva pivotal)

No sistema atual não existe rota de descarte — diretiva do Marcelo (2026-07-05): "não tem desqualificação… pagou é cliente… nunca rejeitar venda". O QCR roteia a OFERTA (qual plano), nunca aceita/rejeita. Falso-descarte =:

  1. Resposta que rejeite ou desestimule a venda (reject_sale);
  2. [ESCALAR_HUMANO]/[HANDOFF_HUMANO] por perfil ou tamanho de carteira (regra 6: JAMAIS);
  3. [PASSAR_BIA] pra quem já decidiu ou carteira pequena (regra 8: Duda fecha).
Caso #1 do golden set = a falha real de 2026-07-04: Duda desqualificou lead de "50 clientes" na 2ª mensagem sem perguntar ticket. Esse caso nunca mais passa.

CREATE TABLE public.agent_eval_results (

4. Runner

Pré-requisito de refactor (mesmo padrão do scenario-match.ts do cakto-recovery): extrair de platform-sales-brain/index.ts as funções puras pra brain-core.ts — buildPrompt(), computeQcrScore(), parsing de tags. Edge Function e runner importam do mesmo módulo. Zero mudança de comportamento.
  1. tools/evals/run-golden.ts (Deno) lê evals/sales/golden/*.jsonl;
  2. Por caso: buildPrompt() real → resposta via LiteLLM (mesma tag de prod) → tag (L1) → extractLeadFacts+computeQcrScore (L2) → judge amostrado (L3, reasoning);
  3. Agrega e sai com código: false_discard>0 → exit 1 · route<0.90 → exit 1 · facts<0.85 → exit 1. Grava run no banco se houver credencial.

CI — .github/workflows/eval-conversao.yml: dispara em PR tocando platform-sales-brain/**, evals/sales/** ou hermes-skills/**; secrets LITELLM_*; exit≠0 bloqueia merge; sumário vira comment. Custo/run: centavos (~50 casos, volume + judge amostrado). Local: deno task eval:golden.

5. Integração com a auto-melhoria

5.1 cerebro-infra (KVM4 — paths reais via SSH)

O gate do reindex (/opt/stacks/cerebro-infra/reindex/run_reindex.sh → eval_gate.py, RAG_EVAL_THRESHOLD=0.7, RAG_EVAL_K=10) continua medindo retrieval — ele gate-ia o ÍNDICE, e isso está certo. A conversão pluga por 3 pontos:

  1. Corpus vencedor: novo dir sources/workstreams/sales/ recebe JSONL semanal das mensagens vencedoras (opt_in/demo sem report) — o reindex diário (cron 05:00 UTC, swap atômico + rollback) absorve. Auto-melhoria real e gated do conhecimento de venda.
  2. Golden queries de conversão no live-check pós-swap do reindex-host.sh.
  3. Fix encontrado na inspeção: gate pré-swap usa K=10, live-check usa k=6 — alinhar em 10 (1 linha).

5.2 Hermes agent (pós-P1 — é ELE o auto-improver, não o openclaw)

  • Skills de venda = arquivos versionados em git (hermes-skills/, agentskills.io); prod monta read-only;
  • Auto-edição em sandbox (backend Docker) → PR → CI roda golden set → merge humano (fase inicial); auto-merge só no P6;
  • Modelo do Hermes → LiteLLM (reasoning na mensagem, volume no resto);
  • command approval + container isolation ligados desde o dia 1.
