1. TL;DR — o veredito

Você pediu três coisas automatizadas via API: gerir números, gerir faturas, pedir números novos. O resultado honesto:

EixoAutomatizável via API?Prova
Pedir número novo✅ 100% APIPOST /api/v2/virtual-phone-accounts — só exige areaCode (DDD); devolve phoneNumber + status na hora
Gerir números✅ 100% APIlist · get · update · cancel · redirect · ler SMS · consultar DDDs
Gerir faturas❌ NÃO existe APIZero endpoints de billing no spec. Faturas são feature só do dashboard

Resposta direta: 2 dos 3 objetivos são totalmente automatizáveis hoje. Faturas não — a Salvy não expõe nenhuma API de billing, e isso é decisão de produto deles, não uma limitação a contornar com código. Para faturas, a saída mais elegante é manter seu próprio ledger: já que você controla o provisionamento via API, você sabe o custo esperado sem precisar da fatura deles.

✅ Confirmação empírica (spike read-only, 2026-07-06)

Rodei o fluxo com a key real (read-only, custo zero) e confirmei ao vivo:

  • Auth Bearer → GET /api/v2/virtual-phone-accounts = 200, 2 linhas reais ("Salão Teste" pending, "Vendas" active).
  • Leitura de OTP funciona E vem estruturada: a linha "Vendas" recebeu "Your WhatsApp code: 944-591" e a Salvy já entrega o código parseado em detections.whatsapp.verificationCode: "944591" — sem regex. Eleva o onboarding WhatsApp de provável para comprovado.
  • status distingue pending de active — máquina de estados do onboarding.
  • area-codes devolve { areaCode, available } por DDD → checa disponibilidade antes de provisionar.
O único eixo ainda não exercido ao vivo é o POST create (billable) — travado no spike atrás de SALVY_ALLOW_CREATE=1, aguardando OK explícito.

2. Inventário completo da API (as 17 rotas)

Base URL: https://api.salvy.com.br · Auth: Bearer token em todas.

Números virtuais — v2 é a versão atual (o núcleo do que interessa)

MétodoRotaO que faz
POST/api/v2/virtual-phone-accountsCriar número (pedir linha). Req: areaCode · opc: name, employeeId, costCenter, customFields
GET/api/v2/virtual-phone-accountsListar todos
GET/api/v2/virtual-phone-accounts/{id}Detalhe de um número
PATCH/api/v2/virtual-phone-accounts/{id}Atualizar (nome, centro de custo, campos)
DELETE/api/v2/virtual-phone-accounts/{id}Cancelar número
GET/api/v2/virtual-phone-accounts/area-codesDDDs disponíveis em estoque
GET/api/v2/.../{id}/sms-messagesLer SMS recebidos — é aqui que sai o OTP
Existem versões v1 e /api/... deprecated das mesmas rotas. Usar sempre v2.

Colaboradores: GET /employees · POST /employees/sync.   Equipamentos: CRUD + archive + timeline-notes.   Webhooks: sms.received — único evento existente.

3. O que a CREATE devolve (por que "pedir número" é trivial)

POST /api/v2/virtual-phone-accounts
Authorization: Bearer <key>
{ "areaCode": 11, "name": "Salão X - Tenant 123", "costCenter": "beauty" }

200 → { id, name, phoneNumber, status, createdAt, canceledAt,
        cancelReason, costCenter, employeeId, customFields }

Você manda um DDD e recebe o número real de volta na mesma resposta. Não há etapa de aprovação/pagamento/estoque no contrato — o custo entra na fatura mensal. 1 chamada = 1 número pronto.

4. O achado estratégico: onboarding de WhatsApp por tenant

No dashboard logado, a linha de teste "Salão Teste - Nexvy" (11) 9XXXX-XXXX exibia "Confirmação do WhatsApp — insira este código na configuração do seu número na API da Meta". O fluxo de negócio que isso habilita:

Venda de tenant (webhook Cakto/Stripe)
  → Edge Function server-side:
     1. POST /api/v2/virtual-phone-accounts { areaCode }      → número novo
     2. Registra número na Meta WhatsApp Business API
     3. Meta envia OTP por SMS → chega no número Salvy
     4. GET /.../{id}/sms-messages (ou webhook sms.received)  → lê o OTP
     5. Confirma na Meta com o OTP                            → WhatsApp ativo

Fecha ponta-a-ponta, sem intervenção humana. É o prêmio real da integração — mais valioso que "gerir faturas". Números Salvy são receive-only, exatamente o suficiente pro OTP da Meta.

5. Limitações a saber ANTES de arquitetar

LimitaçãoImpacto
❌ Sem envio de SMSNúmeros só recebem. Se algum SaaS precisar disparar SMS, a Salvy não serve — precisa Twilio/Zenvia
❌ Sem voz / sem internetLinhas dados-zero, SMS-inbound-only. Sem ligação, sem eSIM de dados
❌ Sem API de faturasConfirmado no spec E no painel. Billing 100% dashboard
⚠️ Um único webhooksms.received apenas. Sem notificação de provisionamento/cancelamento/fatura → estado por polling ou você é a fonte da verdade
⚠️ Rate limit não documentadoSpec não declara 429. Trate como desconhecido → backoff conservador
⚠️ Key exibida uma vez sóIgual Seção 11.1 — gere server-side, cofre, nunca reexibível

6. Faturas: as 4 saídas (e a recomendada)

  • (a) Painel-only — humano confere no dashboard. Simples, mas não é "tudo automatizado".
  • (b) Scraping do dashboard — ❌ NÃO recomendo. Frágil, quebra a cada deploy deles; é gambiarra (contra Seção 3.2).
  • (c) Pedir à Salvy API de faturas / webhook invoice.issued. Caminho certo a médio prazo — vale abrir o chamado.
  • (d) ✅ RECOMENDADA — seu próprio ledger. Você provisiona/cancela via API → tem o registro canônico de números ativos por tenant. Calcule o custo internamente e use a fatura Salvy só p/ reconciliação. Você não precisa da API de faturas pra automatizar o negócio — só pra conferência contábil, que é mensal e tolera ser semi-manual.

Frontend ──JWT Supabase──► Edge Function proxy ──Bearer (server)──► Salvy API
(tenant)                   (valida JWT, chama                       api.salvy.com.br
                            Salvy com a key)
                                  │ escreve
                                  ▼
                          ledger próprio (DB)  ← fonte da verdade p/ faturas
                          números × tenant ×      (reconcilia c/ fatura Salvy)

  • Key Salvy só no .env server-side / cofre — nunca NEXT_PUBLIC_* (Seção 11.1).
  • Todo acesso do front via Edge Function proxy com JWT Supabase.
  • Webhook sms.received → Edge Function grava OTP → dispara confirmação Meta.
  • Cada create/cancel grava no ledger próprio → base de reconciliação.

8. Critério de "pronto" (verificável — Seção 8.3)

  • POST criar número → 200 + phoneNumber → check: número aparece no dashboard
  • GET .../sms-messages lê OTP real da Meta → check: OTP capturado sem humano
  • Key ausente do bundle → check: grep -r salvy_ dist/ vazio
  • Ledger registra 1 número → check: reconcilia com 1 item na fatura Salvy do mês

9. Perguntas abertas

  • Tarifa por número — custo unitário mensal? (define o ledger e o gate de "vale provisionar automático")
  • Salvy tem API/webhook de faturas no roadmap? (decide entre saída c e d)
  • Rate limit real do create — números/minuto? (provisionar em lote sem block)
  • Portabilidade — portar número existente ou só números novos do estoque? (sem endpoint no spec → provável fluxo comercial)
  • Escopo — números p/ WhatsApp Business por tenant (minha leitura), ou também 2FA/SMS de outros fluxos?

10. Próximo passo sugerido
