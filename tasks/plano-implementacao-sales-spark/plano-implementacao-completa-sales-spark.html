5. Decisão de arquitetura — schema canônico vs vertical

O NexvyOficinas usa schema vertical de oficina (empresas/clientes/veiculos/ordens_servico). O sales-spark usa schema genérico CRM (organizations/profiles/leads/deals). Os 81 edges + 54 funções + 377 policies assumem o schema genérico, ancorado em has_role(uuid, app_role) + organization_id.

RotaDescriçãoPrósContras
A — Adotar canônico ⭐Aplicar schema sales-spark completo como base. Domínio oficina vira extensão vertical.81 edges + 54 funções + RLS sem reescrita. Multi-app trivial.Reconciliar inbox já feito (empresas→organizations).
B — Manter oficina + portarManter empresas/etc. Trocar organization_id→empresa_id em 139 tabelas + reescrever 377 policies + 54 funções.Preserva 100% do trabalho atual.Altíssimo risco/esforço. Inviável para 100%.
C — Híbrida (adapter)Adotar canônico + camada de views expondo empresas como view de organizations.Núcleo intacto + código atual via adapter.Complexidade de manutenção; views limitam RLS.

Recomendação: Rota A. É a única que torna "migração 100%" viável no prazo — herda 81 edges + 54 funções + 377 policies prontos em vez de reescrevê-los. O domínio oficina (~6 tabelas) é re-pendurado sobre organizations. Decisão a validar antes da Fase 1.

6. Estratégia multi-app (5 SaaS)

Cada SaaS tem seu próprio project Supabase. Para 100% em todos sem duplicar 490 componentes 5×:

SaasPlugin_vite/
├── packages/
│   ├── core-schema/     ← baseline sales-spark (139 tabelas)
│   ├── core-ui/         ← ~490 components + 121 hooks compartilhados
│   └── core-edges/      ← 81 edge functions
├── apps/
│   ├── NexvyOficinas/   ← core-* + vertical oficina (clientes/veiculos/ordens)
│   ├── NexvyBeauty/     ← core-* + vertical beauty

│   ├── NexvyFoods/      ← core-* + vertical food
│   └── NexvyGYM/        ← core-* + vertical academia

Transforma "implementar 100% em 5 SaaS" em "implementar 100% no núcleo 1× + 5 camadas verticais finas".
